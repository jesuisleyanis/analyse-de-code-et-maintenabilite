Installation avec Docker Compose

Concepts clés

  • Persistance : Utilisation d'une base PostgreSQL pour conserver vos données.
  • Volumes : Stockage des plugins et des logs en dehors du conteneur.
  • Isolation : Communication interne sécurisée entre Sonar et sa base de données.

Objectif

Déployer une instance SonarQube fonctionnelle accessible sur http://localhost:9000.

Attention — Configuration Système

🐧 Linux & WSL2 (Windows) :

Elasticsearch nécessite une limite de mémoire virtuelle plus élevée. Lancez cette commande sur votre hôte (ou dans votre terminal WSL) :

sudo sysctl -w vm.max_map_count=524288
🍎 macOS (Unix) :

Il n'y a généralement pas de configuration système requise. Assurez-vous simplement que Docker dispose d'au moins 4 Go de RAM dans ses réglages.

🪟 Windows (Docker Desktop) :

Si vous utilisez le moteur WSL2, la commande Linux ci-dessus s'applique. Si SonarQube ne démarre pas (logs "max virtual memory areas... too low"), ouvrez un PowerShell et lancez :

wsl -d docker-desktop -u root sysctl -w vm.max_map_count=524288

Validation

  1. 1. Lancez la stack : docker compose up -d
  2. 2. Vérifiez que les conteneurs sont Up
  3. 3. Accédez à l'interface via votre navigateur.

3

Scanner votre projet

Objectif

Lier votre projet RPG à SonarQube et lancer une première analyse complète sans rien installer sur votre machine hôte.

Étape 3.1 — Préparation dans Sonar

Créer un projet local avec la clé rpg-game. Puis générez un Token d'analyse.

Étape 3.2 — Exécution du Scanner (via Docker)

Comme vous n'avez pas forcément Maven installé localement, vous allez construire la commande docker run qui permet de compiler votre projet et d'envoyer les résultats à SonarQube — le tout sans rien installer sur votre machine.

Votre mission

Construisez et exécutez la commande Docker appropriée depuis la racine de votre projet RPG. Elle doit :

  • Utiliser une image Maven avec Java 25 (le projet cible Java 25)
  • Monter votre code source dans le conteneur
  • Permettre au conteneur d'accéder à SonarQube sur votre machine
  • Lancer la compilation, les tests et l'analyse Sonar
  • Passer les paramètres de connexion à votre instance SonarQube

Indices — construire la commande

  • Image : cherchez sur Docker Hub une image officielle maven incluant le JDK 25. Le tag suit la convention maven:X.X-eclipse-temurin-25-....
  • Volume : l'option -v permet de monter un dossier local dans le conteneur. Réfléchissez à comment passer le chemin courant dynamiquement.
  • Réseau : le conteneur doit pouvoir joindre localhost:9000 sur votre hôte. Quelle option Docker permet de partager le réseau de l'hôte ?
  • Goal Maven : le but est de compiler, exécuter les tests ET lancer l'analyse Sonar en une seule commande. Consultez la documentation du plugin sonar-maven-plugin.
  • Paramètres Sonar : Maven accepte des propriétés système via -Dclé=valeur. Quelles propriétés le scanner Sonar attend-il pour se connecter ?

Problèmes courants — que faire si ça ne marche pas ?

Java 25 — UnsupportedOperationException

Java 25 restreint l'accès par réflexion aux modules internes de la JVM. Le plugin Sonar va forcément échouer avec cette erreur si vous n'ouvrez pas explicitement les modules nécessaires. Cherchez comment passer des options JVM à Maven via la variable MAVEN_OPTS et le flag --add-opens.

Mot-clé : MAVEN_OPTS --add-opens java.base ALL-UNNAMED

Windows / Git Bash — "the working directory is invalid"

Git Bash transforme automatiquement les chemins Unix en chemins Windows. Cherchez comment désactiver cette conversion de chemin dans Git Bash via une variable d'environnement à préfixer à votre commande.

Mot-clé : MSYS_NO_PATHCONV

macOS / Windows sans --network host

Sur macOS et Windows, Docker tourne dans une VM : localhost dans le conteneur ne pointe pas vers votre machine. Cherchez l'adresse DNS spéciale fournie par Docker Desktop pour joindre l'hôte.

Validation

Une fois le scan terminé (BUILD SUCCESS), rafraîchissez SonarQube. Votre projet doit afficher ses premiers résultats.

4

Correction des issues

Objectif

SonarQube a détecté 14 issues dans votre projet. Corrigez-les toutes dans le code, relancez le scanner, et atteignez 0 issue ouverte.

Vous êtes libres de corriger le code existant ou de réécrire entièrement les parties qui vous semblent trop difficiles à améliorer — tant que le comportement de l'API reste identique.

Filet de sécurité — Tests unitaires

Pour vous permettre de modifier ou réécrire le code en toute confiance, un fichier de tests unitaires couvrant 100 % des comportements de l'API a été fourni dans votre projet :

src/test/java/com/epsi/rpg/controller/GameControllerTest.java

Ces tests vérifient chaque endpoint et chaque cas particulier (joueur mort, ennemi tué, montée de niveau, boss COOP, mode PVP, items...). Exécutez-les à chaque modification pour détecter immédiatement toute régression :

Règle d'or : si tous les tests passent après votre modification, vous n'avez pas introduit de régression. Si un test échoue, votre correction a cassé quelque chose — lisez le message d'erreur pour comprendre lequel.

Marche à suivre

  1. Ouvrez l'onglet Issues dans SonarQube et parcourez chaque issue.
  2. Pour chaque issue, cliquez sur "Why is this an issue?" pour comprendre ce que Sonar reproche.
  3. Corrigez le code — ou réécrivez la section concernée si nécessaire.
  4. Relancez les tests pour vérifier qu'aucune régression n'a été introduite.
  5. Répétez jusqu'à ce que toutes les issues soient corrigées, puis relancez le scanner Sonar.

Validation

Le TP est terminé lorsque :

  • Le dashboard SonarQube affiche 0 issue ouverte
  • Tous les tests unitaires passent (BUILD SUCCESS)

Faites une capture d'écran du dashboard comme preuve.

5

Quality Gate

Concept

Un Quality Gate est un ensemble de conditions que votre projet doit satisfaire pour être considéré comme prêt à être livré. C'est la porte de sortie qualité : si le projet ne passe pas, le pipeline CI/CD peut être bloqué automatiquement.

Échec

Le projet ne respecte pas les seuils définis. En CI/CD, le déploiement est bloqué.

Succès

Toutes les conditions sont remplies. Le code peut passer en production.

Personnalisable

Chaque équipe définit ses propres seuils selon ses exigences qualité.

Étape 5.1 — Explorer le Quality Gate par défaut

À faire

  1. Dans SonarQube, allez dans Quality Gates (menu principal).
  2. Ouvrez le gate "Sonar way" — c'est le gate par défaut.
  3. Lisez les conditions définies.

Étape 5.2 — Créer un Quality Gate personnalisé

À faire

  1. Créez un nouveau Quality Gate nommé "RPG Gate".
  2. Ajoutez les conditions suivantes :
Métrique Opérateur Seuil
Issues est supérieur à 0
Coverage est inférieur à 50%
Duplicated Lines (%) est supérieur à 3%
  • Appliquez ce gate au projet rpg-game via Project Settings → Quality Gate.

    • Piste Coverage : pour que SonarQube mesure la couverture de code, Maven doit générer un rapport JaCoCo. Cherchez comment configurer jacoco-maven-plugin dans votre pom.xml et ajouter sonar.coverage.jacoco.xmlReportPaths à votre commande de scan.

    Étape 5.3 — Observer et analyser

    À faire

    1. Relancez le scanner avec le plugin JaCoCo configuré.
    2. Observez le résultat du Quality Gate sur votre dashboard.
    3. Identifiez quelle(s) condition(s) font échouer le gate.

    Validation finale du TP

    • Le Quality Gate "RPG Gate" est configuré et appliqué au projet
    • Le dashboard affiche 0 issue et le gate passe sur cette condition
